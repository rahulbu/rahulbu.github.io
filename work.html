GenAI Analytics Assistant

Software Engineer 2 → Senior Software Engineer · 2024 – present · owner of multiple core subsystems

A conversational analytics product: ask a question in plain English and get back SQL, insights, charts, generated apps, or a spoken answer. Under the hood it's a Python multi-agent system built on LangGraph, where an orchestrator plans a request and routes it to specialised agents that share a common LLM, retrieval, and observability foundation.

An orchestrator routes a question to specialised agents, all sharing a provider-agnostic LLM fleet, a RAG vector store, and tracing.

Problem

Analysts want answers from their data in plain English — not dashboards to configure or SQL to write. The hard part isn't calling a model; it's reliably turning a fuzzy, multi-step question into correct, fast analysis over large datasets — and knowing when the question is ambiguous.

Orchestration

  • A LangGraph orchestrator plans each request and routes to the right agent(s), with workflow routing/approval and guardrails on which workflows can run.
  • Parallelised the orchestrator — web-search, dimension-filter extraction, and routing run concurrently to cut end-to-end latency.
  • A dedicated time-filter architecture combines an LLM-backed retriever with pattern matching to resolve date/时间 ranges accurately.

Reliability & ops

  • Health endpoints, index cleanup, and business-view cache eviction for correctness under live data changes.
  • Surfaced actionable error messages instead of opaque failures; added tracing across every agent run.
  • Live-data insight support and export paths (e.g. presentation export).

Agentic App Generation

Senior Software Engineer · primary owner · current focus

The largest and most recent body of my work: an agentic system that generates interactive, editable data applications (React/TSX) from a natural-language prompt — then supports conversational editing of the result. It's a code-generation problem wrapped in an agent loop, with a lot of engineering to make generation reliable and edits cheap.

A prompt becomes a working app through a build → validate → fix loop, then supports conversational, diff-based edits.

How generation works

  • A chat-driven flow gathers context via a context node and a shared design system, then produces a plan/blueprint before any code.
  • Modular, content-addressed code generation — git-like code-module rows with a per-version manifest, replacing a non-scalable monolithic approach.
  • An agentic build → validate → fix loop: the app is compiled, validated, and re-generated on failure until it renders.

Editing without breaking the bank

  • Progressive-disclosure, read-before-write edits — target only the section being changed so small edits send minimal context.
  • Diff-based & JSON-patch edits to cut per-edit LLM tokens.
  • Migration path to convert older monolithic apps into code modules on first edit.

Lifecycle & platform

  • Full versioning — switch, overwrite, delete, archive/unarchive, and version pruning.
  • Async APIs for app creation and data refresh; scheduled data refresh.
  • Multi-view (multi-dataset) support with consistent version handling.

Extensibility

  • A two-layer skills system (implementation vs. behavioural) injected across the planner, context-gatherer, and code-gen stages, with pre-loaded domain/global skills.
  • File and image upload support; data-compatibility checks and "ask-user" clarification.
  • In-flight: interactive, in-app data exploration spanning the agent, renderer, and frontend.

Text-to-SQL & Semantic Query

Software Engineer 2 → Senior Software Engineer · 2025

Turning a question into correct SQL is less about the model and more about structure: first resolve the question into an unambiguous analytic specification, then generate SQL, then execute it safely — learning from prior queries along the way.

Question → semantic spec → SQL → execute, with a retry loop and a RAG query-seed store feeding learned examples back in.

Semantic layer first

  • A structured layer converts a question into an analytic spec — aggregations, filters, comparisons, group-by, HAVING, top/bottom, time resolution — before any SQL.
  • Handles multi-comparison, market-share filters, enum-based aggregation, and case-insensitive matching; normalises dimension-filter operators (contains / not-equal / like).
  • Asks the user to clarify when a request is genuinely ambiguous.

Learning from past queries

  • Query-seed learning — index seed queries and their variations into a vector store, then retrieve learned examples for new questions (RAG).
  • A learning loop keeps improving coverage as more queries are seen.

Robust SQL generation

  • Charting output, a clarification flow, and retry on execution failure.
  • ClickHouse dialect support, alias/column handling, CTE correctness, and fact-to-fact join guards.

Fast & tunable

  • An LLM response cache avoids repeat calls; per-stage multi-LLM assignment.
  • Prompt-variation support and a debug mode; exposed/configurable prompts.
  • Net effect: ~30% lower query latency via context optimization and caching.

Real-time Voice AI

Senior Software Engineer · 2025 – 2026

Real-time, spoken conversation with the analytics assistant — the kind of feature that demos in minutes but only survives real traffic with the right backend plumbing.

The agents

  • Integrated multiple providers behind one pattern: a Deepgram voice agent (with a switch to its low-latency model), an ElevenLabs agent over WebRTC, and OpenAI Realtime.
  • Tools, prompts, and ASR keywords hot-sync on startup and per session — config resolves from middleware with an environment fallback.

The infra detail that mattered

  • WebSockets don't work on the default web-server worker — I moved the service to a gevent worker model.
  • Tuned concurrency to ~500 concurrent voice sockets, and improved audio playback smoothness.

Observability

  • LangSmith tracing across voice tool runs to debug real conversations.
  • Fixed voice-mode context handling so spoken sessions carry the same reasoning as text.

Multi-LLM / Provider Abstraction

SE2 → Senior Software Engineer · 2025, ongoing

A GenAI product shouldn't be married to one vendor, or need a redeploy to change a model. I built the provider-agnostic layer the whole assistant runs on.

The abstraction

  • A unified config abstraction over Anthropic Claude, OpenAI, Google Gemini/Vertex, AWS Bedrock (and Azure OpenAI).
  • Per-stage model assignment — each step of the pipeline can use the model best suited to it.

Hot config, no restart

  • Real-time config sync via a message-queue (RabbitMQ) listener — model/assignment changes reset and reload models live, without a redeploy.
  • A config-validation API with structured output catches bad config before it ships.

Reliability

  • Blind retry with failure-cache eviction so a transient failure doesn't poison the cache.
  • Thread-safe LLM caching; structured-output support across providers.
  • Led a major LangChain / LangGraph version upgrade across the codebase.

Search & Ranking Platform

SE1 → Senior Software Engineer · ~5 years of ownership

For nearly five years I owned doctor/clinic discovery — the indexing platform, the query path, autocomplete, and the ranking that decides who shows up first. It's also where the search→AI throughline begins: retrieval, ranking, and relevance are exactly the muscles that make RAG and Text-to-SQL work later.

Five years evolving discovery — from an indexer, to realtime bucketised ranking, to ML learning-to-rank.

Indexing platform

  • Built and containerized the search-indexer (Spring Boot, Elasticsearch); migrated its build Gradle→Maven.
  • Realtime, event-based sync plus one-time (re)indexing and environment-specific workers.
  • Elasticsearch multi-search queries, index-refresh handling, and partial updates.

Autocomplete

  • Migrated autocomplete from Lucene to Elasticsearch, with realtime indexing and partial updates.
  • Continually tuned autocomplete queries for relevance and speed.

Ranking evolution

  • Hyperlocal listing weights → distance/availability bucketisation → configurable bucketed weights.
  • Then ML learning-to-rank: built the feature set, the model, and an LTR store, integrated via an Elasticsearch LTR plugin.

Reach & growth

  • Powered discovery for millions of patient searches; launched Listings & Profiles in the UAE.
  • Google Feeds and sitemap generation for organic discovery; establishment data sync and merge-listing correctness.

Practo · Health-tech Platform

ABDM National Health-Data Integration

Software Engineer 2 → Senior Software Engineer · 2023

India's Ayushman Bharat Digital Mission (ABDM) is a national, consent-first health-data network built on FHIR R4. Integrating a large platform into it — as both a provider and a consumer of records — is a serious interoperability and compliance effort.

Records only flow on patient consent: the platform requests via the consent manager, the patient approves, and the provider returns a FHIR R4 bundle.

What I built

  • HIP and HIU APIs — the platform acts as both a health-information provider and user.
  • Consent flows and consent-managed data transfers between HIP and HIU.
  • FHIR R4 parsers to produce and consume compliant health-record bundles.

Why it's hard

  • Records move only on a patient's informed consent — a consent state machine, not a simple API call.
  • Requires patient identity (ABHA) and practitioner/facility registry integration, with strict compliance.
  • Interoperability with a national standard means the parsers must be exactly right.
